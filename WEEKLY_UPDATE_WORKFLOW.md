# 내수동교회 모바일 주보 주간 업데이트 절차

이 문서는 매주 새 주보 이미지를 받아 모바일 주보 서비스로 업데이트하는 표준 절차를 정리합니다.

## 목표

새 주보 이미지 또는 PDF를 바탕으로 주보 내용을 구조화하고, 모바일 주보 UI에 맞춰 업데이트한 뒤 검수와 승인 절차를 거쳐 GitHub Pages에 배포합니다.

## 주간 작업 흐름

1. 최신 주보 자동 확인
   - 기본 확인 명령은 `npm run bulletin:check`입니다.
   - 스크립트가 `https://www.naesoo.or.kr/Board/Index/77077`의 최신 게시물 제목에서 날짜를 추출합니다.
   - 최신 게시물이 현재 `archive.json`의 `current`보다 새 주보일 때만 다운로드 대상으로 판단합니다.
   - 이미 현재 주보와 같은 날짜라면 `status=already-current`로 멈춥니다.

2. 원본 주보 자동 다운로드
   - 새 게시물이 감지되면 `npm run bulletin:fetch`를 실행합니다.
   - 게시물 상세 페이지의 이미지 또는 첨부파일을 확인해 `original/YYMMDD/` 폴더에 저장합니다.
   - 현재 주보를 다시 받으려면 `npm run bulletin:fetch:force`를 사용합니다.
   - 다운로드 후 `original/YYMMDD/manifest.json`에 게시물 URL, 파일명, 처리 다음 단계가 기록됩니다.

3. 원본 주보 수동 전달
   - 채팅창에 주보 이미지를 페이지 순서대로 업로드합니다.
   - 파일명에 페이지 번호가 있으면 그대로 사용합니다.
   - 순서가 불명확하면 “1페이지, 2페이지”처럼 페이지 순서를 함께 알려줍니다.
   - 로컬 폴더나 GitHub에 원본을 올린 경우에는 경로 또는 링크를 알려줍니다.
   - 자동 다운로드가 실패했거나 교회 홈페이지 게시물이 아직 올라오지 않았을 때만 수동 전달을 사용합니다.

4. 내용 추출
   - 이미지에서 텍스트를 추출합니다.
   - 다음 항목을 중심으로 구조화합니다.
     - 발행일, 권호, 표어
     - 주일예배 순서
     - 주중예배
     - 교회소식
     - 주일설교요약
     - 구역교회 순서와 질문
     - 예배 및 모임 안내
     - 섬기는 사람들
     - 봉사위원
     - 오시는 길과 주차 안내

5. JSON 데이터화
   - 추출한 내용을 `bulletins/YYYY-MM-DD.json` 형식으로 저장합니다.
   - `archive.json`의 `current`와 `issues`에 새 주보를 등록합니다.
   - 예배 1부~3부 내용이 동일한 경우에는 묶어서 표시되도록 정리합니다.
   - 날짜, 시간, 장소, 이름, 찬양 제목은 원본과 대조합니다.

6. 모바일 주보 빌드
   - 저장소 루트에서 `npm test`를 실행합니다.
   - 빌드 도구가 현재 주보 JSON을 읽어 `index.html`, PWA 아이콘과 서비스 워커를 생성합니다.
   - 기존 모바일 UI, 탭 구조, 검색, 공유, PWA 설정은 유지합니다.
   - 새 주차 내용만 교체합니다.

7. 검수
   - 오탈자, 띄어쓰기, 맞춤법을 확인합니다.
   - 날짜와 요일이 맞는지 확인합니다.
   - 이름 표기, 직분, 장소, 시간, 찬양 제목을 원본과 대조합니다.
   - 모바일 화면에서 줄바꿈, 간격, 카드 높이, 하단 메뉴 겹침을 확인합니다.
   - 검색 결과에 새 주보 내용이 반영되는지 확인합니다.

8. 로컬 미리보기
   - 로컬 서버에서 `index.html`을 확인합니다.
   - 홈, 예배, 소식, 말씀, 모임, 안내 탭을 모두 확인합니다.
   - 필요하면 Galaxy Fold, Flip 등 미리보기 크기로도 확인합니다.

9. 배포 승인
   - GitHub에 푸시하기 전 반드시 사용자에게 확인합니다.
   - 사용자가 “푸시해”, “배포해”, “진행해”처럼 명확히 승인한 경우에만 푸시합니다.

10. GitHub 배포
   - 승인 후 `main` 브랜치에 커밋과 푸시를 진행합니다.
   - GitHub Pages 배포 브랜치 또는 배포 워크플로도 함께 확인합니다.
   - 공개 URL에서 최신 문구가 반영되었는지 확인합니다.

## 검수 체크리스트

- [ ] 페이지 순서가 원본과 일치한다.
- [ ] 발행일과 권호가 정확하다.
- [ ] 주일예배 1부~3부 묶음이 정확하다.
- [ ] 설교 제목, 본문, 설교자가 정확하다.
- [ ] 대표기도, 찬양대, 찬송, 축도가 원본과 일치한다.
- [ ] 교회소식의 날짜, 요일, 장소가 정확하다.
- [ ] 결혼, 출산, 소천 등 이름 표기가 정확하다.
- [ ] 구역교회 질문과 암송 구절이 정확하다.
- [ ] 예배 및 모임 시간과 장소가 정확하다.
- [ ] 봉사위원 명단이 빠지지 않았다.
- [ ] 오시는 길과 주차 안내가 최신 내용이다.
- [ ] 오탈자, 띄어쓰기, 맞춤법 문제가 없다.
- [ ] 모바일에서 읽기 좋은 줄바꿈과 간격이다.
- [ ] 검색 결과에 새 내용이 반영된다.
- [ ] GitHub 푸시 전 사용자 승인을 받았다.

## 운영 원칙

- 원본 주보의 고유명사와 찬양 제목은 임의로 고치지 않습니다.
- 맞춤법상 어색해 보여도 이름, 찬양 제목, 기관명은 원본 우선으로 처리합니다.
- 명확한 오탈자나 모바일용 문구 다듬기는 검수 후 반영합니다.
- 배포 전에는 항상 로컬 미리보기와 공개 전 승인 절차를 거칩니다.
- GitHub 푸시는 사용자 승인 없이는 진행하지 않습니다.
- 자동 다운로드는 원본 확보 단계까지만 수행합니다.
- JSON 변환과 최종 배포는 검수 후 진행합니다.

## 파일 구조

```text
bulletins/
  2026-06-14.json
  2026-06-21.json
scripts/
  build.js
  fetch-latest-bulletin.js
  generate-icons.js
tests/
  validate.js
archive.json
index.html
```

매주 작업은 “`npm run bulletin:check` → `npm run bulletin:fetch` → JSON 생성 → `archive.json` 갱신 → `npm test` → 미리보기·검수 → 승인 후 배포” 순서로 진행합니다.

GitHub Pages 배포는 `main` 브랜치 푸시를 감지하는 Actions 워크플로 하나로만 처리합니다. 별도의 `gh-pages` 수동 푸시는 하지 않습니다.
